# tech-deep-dive（自律型テック深掘りスキル）

このフォルダは **Claude の Agent Skill** です。スキル名は `tech-deep-dive`（`SKILL.md` の YAML `name` と一致）。旧称や別名として README やデモで `arxiv-research-assistant` と書かれている箇所がありますが、**実体は同一コンセプトの拡張版**と捉えてください。

---

## このスキルは何をするか

ユーザーが **1つの技術テーマ**（例: GraphRAG、MCP、ある論文・リポジトリ）を指定すると、エージェントが Web 上の複数ソースを横断調査し、**分析からドキュメント・サンプル・スライドまで**をまとめて出力します。

向いている用途:

- 新技術のキャッチアップ（論文＋実装＋ブログを一度に整理したい）
- 導入検討（比較軸・実装の成熟度・クイックスタートまで欲しい）
- 社内共有（ブログ風記事・チュートリアル・スライドまで揃えたい）

---

## 誰向けの README か

**スキルを書いた人でなくても**、次が分かるように書いています。

- フォルダの役割と、本体はどのファイルか
- 実際に何が生成されるか（ファイル名・場所・ざっくり分量）
- 制約（日本語・絵文字禁止など）
- 他スキル・ツールへの依存
- トラブル時に見るべき場所

詳細な手順・セクション構成の**正本**は常に [`SKILL.md`](./SKILL.md) です。README は「地図」、`SKILL.md` は「仕様書」です。

## 成果物一覧（6フェーズ）

`SKILL.md` では **6つのフェーズ**が連続実行される想定です。出力はプロジェクト直下の `research/` 以下にまとまります。

**フォルダ命名**: `research/{YYYY-MM-DD}_{テーマのケバブケース}/`

例: `research/2026-03-20_graph-rag/`

| フェーズ | 主な出力ファイル | 内容のイメージ |
|----------|------------------|----------------|
| 1 | `technical_report.md` | 長文の技術調査レポート（全体像・個別ソース解説・比較・用語集など） |
| 2 | `implementation_guide.md` | 選定・環境構築・クイックスタート・本番観点の実装ガイド |
| 3 | `blog_article.md` | Qiita / Zenn 想定の読み物としての技術記事 |
| 4 | `tutorial.md` | ハンズオン手順（ステップ分割・検証・発展課題） |
| 5 | `samples/` | `basic_example` / `practical_example` / `advanced_example` 等の**動くコード**＋ `samples/README.md` |
| 6 | `presentation.pptx` | 共有用スライド（枚数目安 15〜25） |

**注意**: フェーズ6は `SKILL.md` 上 **pptx スキル**の利用が前提に書かれています。実行環境に pptx スキルが無い場合、スライド生成だけ失敗・省略される可能性があります。その場合は `SKILL.md` の該当箇所に合わせて手動でスライド化するか、Markdown だけで共有する運用になります。

## エージェントにどう指示するか

スキルは **説明文（description）** からエージェントが関連付けて読み込む方式です。次のような依頼で発火しやすくなります。

- 「〇〇について **マルチソースで深掘り**して、レポートと実装ガイドまで欲しい」
- 「論文・GitHub・ブログを横断して **技術理解レポートとサンプル**まで一式」
- 「**tech-deep-dive** で △△ を調査して」

プロダクトによってはスラッシュコマンド（例: `/tech-deep-dive`）を別途定義している場合もあります。**このリポジトリの README にあった `/arxiv-research-assistant` は旧例**です。実際に使うクライアントの設定を確認してください。

## 守るべきルール（スキル全体の制約）

`SKILL.md` に明記されている主なもの:

- **成果物の言語**: 日本語（固有名詞・API 名などは英語のまま可）
- **絵文字**: 使用しない
- **コード内コメント**: 日本語

第三者がレビューするときは、上記が守られているかもチェックポイントになります。

---

## 調査・生成の前提

- **Web 検索・フェッチ**が使えること（`WebSearch` / `WebFetch` 等）。オフラインでは成立しません。
- **時間・トークン**: 長文レポート＋複数ファイル＋サンプルコードは負荷が大きいです。1テーマあたり「長めのセッション」想定です。
- **情報の鮮度**: 取得時点の Web に依存します。論文・バージョンは必ず一次ソースで確認してください。
