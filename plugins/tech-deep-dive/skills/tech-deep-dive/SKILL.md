# Autonomous Tech Research Skill

## 概要

このスキルは、ユーザーが指定した一つの技術テーマについて、Web上の複数ソース（arXiv, GitHub, 技術ブログ, 公式ドキュメント, スライド/動画）を横断的に調査し、以下の成果物を一度の指示で自律的に生成する。

- 技術理解レポート -- 技術の全体像と深い分析
- 実装応用ガイド -- 導入・運用のための実践ガイド
- ブログ記事 -- 第三者に共有できる技術解説記事
- チュートリアル -- 手を動かして学ぶステップバイステップ教材
- コードサンプル集 -- すぐに動かせる実装例
- プレゼンスライド -- 社内共有・勉強会用資料

「一度命じたら、あとは任せろ」のコンセプトを追求し、情報収集から分析、資料作成までの全プロセスを自動化する。

## 重要な制約

- **全ての成果物は日本語で作成すること。** スライド、レポート、ガイド、ブログ記事、チュートリアルの全てにおいて、タイトル・見出し・本文・図表のキャプション・注釈を日本語で記述する。技術用語（固有名詞、ライブラリ名、API名等）はそのまま英語表記を維持し、必要に応じて日本語の補足説明を括弧内に付記する。
- **絵文字を一切使用しないこと。** 見出し、本文、スライド、コメント等の全ての箇所で絵文字を使用しない。
- **コード内のコメントも日本語で記述すること。**

## ワークフロー（完全自律実行）

ユーザーの指示を受けると、以下の6つのフェーズを連続して実行し、最後に統合された成果物を提出する。

---

### フェーズ1：技術理解レポート生成（technical_report.md）

テーマに関する全体像と核心を捉えるための、**技術ブログの連載記事に匹敵する深さと分量**の分析レポートを作成する。目安として**3,000〜8,000語（日本語）以上**の本格的な技術記事を目指す。

#### 1.1 調査プロセス

1.  **マルチソース並列調査**: `WebSearch`と`WebFetch`を駆使し、5つの主要ソース（arXiv, GitHub, 技術ブログ, 公式ドキュメント, スライド/動画）から関連情報を並列で収集する。各ソースについて最低3件以上の情報を取得すること。
2.  **情報の統合とクロスリファレンス**: 収集した断片的な情報を統合し、論文で提案された手法がどのGitHubリポジトリで実装され、どの技術ブログで解説されているかといった関係性を紐付ける。
3.  **分野・技術スタックの特定**: 収集した情報から、テーマが属する技術分野と関連する主要な技術スタックを特定する。
4.  **評価軸の動的生成**: `references/evaluation_heuristics.md`を参照し、特定した分野に最適な評価軸を動的に決定する。

#### 1.2 レポート構成（必須セクション）

以下の構成に従い、各セクションを**省略せず詳細に**記述すること。

```
# [テーマ名] 技術調査レポート

## 1. エグゼクティブサマリー
- テーマの概要を3〜5段落で記述
- この技術が注目される背景と動機
- 調査で得られた最も重要な知見を3つ箇条書き

## 2. 背景と問題設定
- この技術が解決しようとしている課題の詳細な説明
- 従来のアプローチとその限界
- なぜ既存手法では不十分なのかの技術的根拠
- この分野の歴史的経緯（主要なマイルストーンを時系列で）

## 3. コア技術の詳細解説
- 技術の基本原理を段階的に解説（前提知識から積み上げる形式）
- アーキテクチャや設計思想の詳細（図やダイアグラムをMermaid.jsで表現）
- 数式やアルゴリズムがある場合は、その直感的な意味も含めて説明
- 具体例を用いた動作の説明（「例えば〜の場合、〜となる」形式）

## 4. 主要な研究・実装の個別解説
- 収集した各ソースについて、以下の形式で個別に詳細解説:
  ### 4.x [ソース名/論文名]
  - **概要**: 1段落の要約
  - **目的**: 何を解決しようとしているか
  - **手法**: どのようなアプローチを取っているか（技術的詳細を含む）
  - **結果・成果**: 定量的な評価結果、ベンチマーク成績
  - **独自の貢献**: 他との差別化ポイント
  - **限界・課題**: 著者自身が認めている制約や今後の課題
  - **出典URL**: 原文へのリンク

## 5. 横断比較分析
- 動的に生成した評価軸に基づく詳細な比較表（Markdown表形式）
- 比較表だけでなく、比較結果の分析を文章で2〜3段落記述
- 各アプローチの強み・弱みの考察
- どのようなユースケースにどのアプローチが適するかの判断基準

## 6. 技術エコシステムと関連技術マップ
- 関連技術スタックの全体像をMermaid.jsダイアグラムで表現
- 各構成要素の役割と相互関係の解説
- 競合技術や代替技術との位置づけ

## 7. 業界動向と採用事例
- この技術を採用している主要企業・プロジェクトの事例
- 業界全体のトレンドと今後の方向性
- マーケットでの位置づけ

## 8. 専門用語集
- レポート中に登場する全ての専門用語について:
  - 用語名（英語表記がある場合は併記）
  - 1〜3文の平易な日本語での定義
  - 関連する用語へのクロスリファレンス

## 9. 次に読むべき資料
- カテゴリ別に整理（入門向け / 実践向け / 最先端研究）
- 各資料について推薦理由を1〜2文で記述
- URLを必ず付記

## 10. 参考文献
- 調査で参照した全ソースをカテゴリ別に整理
- 各参考文献にURLを付記
```

#### 1.3 品質基準

- 単なる箇条書きの羅列ではなく、**文章として読める技術記事**を書くこと。各セクションには導入文と結びの文を含め、セクション間の論理的なつながりを意識する。
- 「〜である」「〜を解説する」のような教科書調ではなく、**「なぜそうなるのか」「何が嬉しいのか」「どこに注意すべきか」**を読者に伝える実践的なトーンで書く。
- 抽象的な説明だけでなく、**具体例やコードスニペット**を積極的に挿入して理解を助ける。
- 図表（Mermaid.js）は最低3箇所以上挿入すること。

### フェーズ2：実装応用ガイド作成（implementation_guide.md）

フェーズ1で得られた理解を元に、**実際に手を動かして技術を導入するための実践ガイド**を作成する。目安として**3,000〜6,000語（日本語）以上**の分量を確保する。

#### 2.1 調査プロセス

1.  **GitHubリポジトリの詳細調査**: フェーズ1で特定した主要なGitHubリポジトリについて、スター数、最終更新日、Issue/PRの活発度、ライセンスなどを調査し、実装の成熟度とメンテナンス状況を評価する。
2.  **環境構築手順の整理**: READMEやドキュメントを読み解き、最も標準的と思われる環境構築手順や依存ライブラリを整理する。
3.  **類似実装との比較と推奨構成の提示**: 複数の実装が存在する場合、それぞれの長所・短所を比較し、ユーザーの目的に合わせた推奨構成を提示する。

#### 2.2 ガイド構成（必須セクション）

```
# [テーマ名] 実装応用ガイド

## 1. はじめに
- このガイドの目的と対象読者
- 前提知識・スキルの明示
- ガイド全体の構成案内

## 2. 技術選定ガイド
- 利用可能な実装・ライブラリの一覧表
  （名前、言語、スター数、最終更新日、ライセンス、メンテナンス状況）
- ユースケース別の推奨選定:
  - プロトタイプ・検証目的の場合
  - 本番環境・プロダクション利用の場合
  - 学習・研究目的の場合
- 選定理由の詳細な解説

## 3. 環境構築
- 前提条件（OS、ランタイムバージョン等）
- ステップバイステップのセットアップ手順（コマンド付き）
- 依存ライブラリの一覧と各ライブラリの役割説明
- よくあるインストールエラーとその対処法

## 4. クイックスタート
- 最小構成で動作するサンプルコード（コピー&ペーストで動くレベル）
- コードの各パートの詳細な解説（何をしているか、なぜそうするか）
- 実行結果の例示と解説

## 5. 実践的なユースケースと応用例
- 3つ以上の具体的なユースケースを想定し、それぞれについて:
  - ユースケースの説明
  - 実装アプローチの解説
  - サンプルコード（動作可能なレベル）
  - 期待される結果と評価方法

## 6. 設計パターンとベストプラクティス
- 推奨されるアーキテクチャパターン（図解付き）
- アンチパターン（やってはいけないこと）とその理由
- パフォーマンスチューニングのポイント
- スケーラビリティに関する考慮事項

## 7. トラブルシューティング
- よくある問題とその解決策（FAQ形式）
- デバッグのためのツールやテクニック
- パフォーマンス問題の診断方法

## 8. 本番環境への導入ガイド
- プロダクション化に際して考慮すべきポイント
- セキュリティに関する注意事項
- 監視・運用のベストプラクティス
- コスト見積もりの考え方（該当する場合）

## 9. 参考リソース
- 公式ドキュメントへのリンク
- コミュニティリソース（Discord, Slack, フォーラム等）
- 関連ツール・ライブラリ
```

#### 2.3 品質基準

- 全てのコードスニペットは**そのままコピー&ペーストで動作する**レベルの完成度を目指す。省略（`...`）を最小限にする。
- コードには**日本語のコメント**を付与して、各行・各ブロックが何をしているか説明する。
- 「動かすだけ」ではなく**「なぜそうするか」「他の選択肢と比べてなぜこれか」**を必ず説明する。
- 環境構築手順は**初学者でも迷わないレベル**の粒度で記述する。

### フェーズ3：ブログ記事生成（blog_article.md）

調査結果を、**Qiita や Zenn に投稿できるレベル**の技術ブログ記事として再構成する。技術理解レポートの要約ではなく、読者の興味を引き、最後まで読ませる独立した記事として書く。目安として**2,000〜5,000語（日本語）以上**。

#### 3.1 記事構成（必須セクション）

```
# [キャッチーなタイトル]

## はじめに -- なぜこの技術が面白いのか
- 読者の関心を引くフック（課題提起、驚きの事実、身近な例え）
- この記事を読むと何がわかるか
- 対象読者の想定

## この技術が生まれた背景
- 従来の課題を具体的なシナリオで説明
- 「こんな経験はないだろうか」形式で読者の共感を誘う
- 既存手法の限界を平易に解説

## 技術の核心を理解する
- コアコンセプトを日常的な比喩やアナロジーで説明
- 段階的に複雑さを増していく解説構成
- 図解（Mermaid.js）を交えた視覚的な説明
- 「つまり何が嬉しいのか」を各ポイントで明示

## 実際に動かしてみる
- 最小限のコードで「動く」ことを体験させるデモ
- 実行結果とその解説
- 「ここを変えるとこうなる」という実験的な解説

## 深掘り -- 仕組みを理解する
- コアアルゴリズムやアーキテクチャの詳細
- なぜそのような設計になっているかの技術的根拠
- パフォーマンス特性やトレードオフの考察

## 実践で使うなら -- 現場での活用ポイント
- プロダクション利用での注意点
- よくあるハマりポイントとその回避策
- 他の技術との組み合わせパターン

## まとめと今後の展望
- 記事全体のポイントを3〜5行で振り返り
- この技術の今後の発展方向
- さらに学びたい人向けのリソース案内
```

#### 3.2 品質基準

- **読み物として面白い記事**を目指す。箇条書きの羅列ではなく、段落で語る文章を基本とする。
- 技術的正確性を保ちつつ、**平易な日本語**で書く。専門用語の初出時には必ず簡潔な説明を添える。
- 読者が「自分も試してみよう」と思えるよう、**動かせるコード例**を必ず含める。
- 比喩やアナロジーを効果的に使い、抽象的な概念を直感的に理解できるようにする。
- Mermaid.js の図を最低2箇所含める。

### フェーズ4：ハンズオンチュートリアル生成（tutorial.md）

読者が**ゼロから手を動かして技術を体験できる**、ステップバイステップのチュートリアルを作成する。目安として**2,000〜4,000語（日本語）以上**。

#### 4.1 チュートリアル構成（必須セクション）

```
# [テーマ名] ハンズオンチュートリアル

## このチュートリアルについて
- 所要時間の目安（例：約30分〜1時間）
- 完成時に得られるもの（具体的な成果物の説明）
- 前提条件（必要な知識、ツール、アカウント等）

## ステップ0：環境準備
- 必要なツールのインストール手順（バージョン指定あり）
- 動作確認コマンド
- トラブルが起きた場合の対処法

## ステップ1：[最初の小さなゴール]
- このステップで達成すること
- 具体的な手順とコード
- 実行結果の確認方法
- うまくいかない場合のチェックポイント

## ステップ2：[次のゴール]
- （ステップ1と同様の構成）

## ステップ3：[さらに発展]
- （同様の構成、段階的に複雑さが増す）

...（必要なだけステップを追加。最低5ステップ以上）

## ステップN：完成 -- 全体を統合する
- 全ステップの成果を統合
- 完成したアプリケーション/システムの動作確認
- 完成物のスクリーンショットや出力例

## 発展課題
- チュートリアルの延長として取り組める3〜5つの課題
- 各課題のヒントや方向性（答えは書かない）

## 振り返り -- 何を学んだか
- チュートリアルで身についたスキル・知識の整理
- 次に学ぶべきトピックへの橋渡し
```

#### 4.2 品質基準

- 各ステップは**1つの明確なゴール**に集中する。1ステップで複数のことを同時にやらない。
- 全てのコマンド・コードは**コピー&ペーストで動作する**こと。ファイル名、パス、変数名は一貫させる。
- 各ステップの冒頭に「このステップで何をするか」、末尾に「ここまでで何ができるようになったか」を明記する。
- **期待される実行結果**（コマンド出力、画面表示等）を必ず示す。
- エラーが起きやすい箇所には**事前に警告と対処法**を記載する。
- ステップ間の**依存関係を明確に**する（「ステップ2はステップ1が完了していることが前提」等）。

### フェーズ5：コードサンプル集の生成（samples/ ディレクトリ）

テーマに関する**実際に動作するコードサンプル**を複数作成し、`samples/` ディレクトリにまとめる。

#### 5.1 サンプルの種類（最低3つ以上作成）

1.  **基本サンプル（basic_example）**: 最小限のコードで技術の基本動作を示す
2.  **実践サンプル（practical_example）**: 実際のユースケースに近い、やや複雑な実装
3.  **応用サンプル（advanced_example）**: 複数の技術を組み合わせた発展的な実装

テーマに応じて追加のサンプルを作成してよい（ベンチマーク用、テスト用、統合例など）。

#### 5.2 各サンプルの要件

- **即座に実行可能**であること。依存関係のインストール手順を含む。
- ファイル冒頭に以下を日本語コメントで記載:
  - サンプルの概要（何を示すか）
  - 実行方法（コマンド）
  - 前提条件（必要なライブラリ、環境変数等）
  - 期待される出力
- コード内の重要な箇所に**日本語コメント**で解説を付与する。
- **README.md** を `samples/` ディレクトリに作成し、全サンプルの一覧と説明、実行方法をまとめる。

#### 5.3 ディレクトリ構成例

```
samples/
  README.md                    # サンプル一覧と使い方
  basic_example/
    main.py (or .ts, .js等)    # 基本サンプル
    requirements.txt           # 依存関係（該当する場合）
  practical_example/
    main.py
    config.yaml                # 設定ファイル（該当する場合）
    requirements.txt
  advanced_example/
    main.py
    requirements.txt
```

使用言語はテーマに最も適した言語を選択する（Python, TypeScript, Go等）。

### フェーズ6：プレゼンスライド生成

調査結果を他者に共有するためのプレゼンテーションスライドを自動生成する。

#### 6.1 言語要件

**スライドの全コンテンツを日本語で作成すること。** 具体的には以下の全てを日本語で記述する:
- スライドタイトル
- 見出し・小見出し
- 本文・箇条書き
- 図表のキャプション・ラベル
- スピーカーノート

技術用語（固有名詞、ライブラリ名、API名、コード等）のみ英語表記を維持する。

#### 6.2 生成プロセス

1.  **要点抽出**: これまでの全フェーズの成果物から、最も重要な図表や結論を抽出する。
2.  **スライド自動生成**: `/pptx`スキルを使用し、以下の構成で**15〜25枚**のスライドを生成する:
    - 表紙（テーマ名、調査日）
    - 目次
    - エグゼクティブサマリー（1〜2枚）
    - 背景と課題（2〜3枚）
    - コア技術の解説（3〜5枚、図解を含む）
    - 主要アプローチの比較（2〜3枚、比較表を含む）
    - 実装方法の概要（2〜3枚）
    - デモ/コードサンプルの紹介（1〜2枚）
    - 採用事例・ユースケース（1〜2枚）
    - まとめと次のアクション（1〜2枚）
    - 参考文献（1枚）

#### 6.3 品質基準

- 1枚のスライドに詰め込みすぎず、**1スライド1メッセージ**を原則とする。

## 最終成果物

### 出力先フォルダ

調査結果はテーマごとに専用フォルダを作成し、その中に全成果物を集約する。フォルダはプロジェクトルート直下の `research/` ディレクトリ配下に作成する。

**フォルダ命名規則**: `research/{YYYY-MM-DD}_{テーマ名のケバブケース}/`

例:
- テーマ「GraphRAG」の場合: `research/2026-03-20_graph-rag/`
- テーマ「Model Context Protocol」の場合: `research/2026-03-20_model-context-protocol/`

### フォルダ構成

```
research/{YYYY-MM-DD}_{テーマ名}/
  technical_report.md      # フェーズ1: 技術理解レポート（3,000〜8,000語以上）
  implementation_guide.md  # フェーズ2: 実装応用ガイド（3,000〜6,000語以上）
  blog_article.md          # フェーズ3: 技術ブログ記事（2,000〜5,000語以上）
  tutorial.md              # フェーズ4: ハンズオンチュートリアル（2,000〜4,000語以上）
  samples/                 # フェーズ5: コードサンプル集
    README.md
    basic_example/
    practical_example/
    advanced_example/
  presentation.pptx        # フェーズ6: プレゼンスライド（日本語、15〜25枚）
```

### 実行手順

1. `research/` ディレクトリが存在しない場合は作成する。
2. テーマ名から命名規則に従ったフォルダ名を生成し、フォルダを作成する。
3. 全ての成果物をそのフォルダ内に出力する。
4. 完了時にフォルダパスと各成果物の概要をユーザーに提示する。

## リソース

### `references/evaluation_heuristics.md`

技術分野ごとに、どのような評価軸で技術や実装を比較すべきかの指針を記述したファイル。エージェントはレポート作成時にこのファイルを参照し、比較表の項目を動的に決定する。
